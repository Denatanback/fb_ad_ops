# Step 27 — UI/UX Audit

**Date:** 2026-03-20
**Type:** Audit-only pass. No code changes.
**Scope:** Full workspace — shell, sidebar, topbar, dashboard, imports, admin pages, creatives

---

## 1. Overall Assessment

The product has a well-reasoned data model and a working design system (CSS variables, consistent components, good badge language). However, **every page is built from the same template**, and that template overemphasizes explanation over operation.

The recurring pattern:
```
PageHeader (eyebrow + h2 + description)
stats-inline (3–4 stat cards with paragraph explanations)
list-layout (2 × SectionCard side by side)
SectionCard (table or form)
```

This makes every screen feel like the same screen in a different color. An operator looking at the imports page and then the notifications page gets an identical structural experience — which means neither page has clear identity or fast scanning paths.

The core problem: **the UI explains itself too much and surfaces too little operational signal at a glance**.

---

## 2. Main Structural Problems

### P1 — Triple title hierarchy
Every page has three levels of "this is what this page is":
1. Sidebar active item (e.g., "Импорты")
2. Topbar h1 driven by navigation config (same: "Импорты")
3. PageHeader with eyebrow + h2 + description inside the page body

The user reads the same context three times before reaching any content. The topbar is doing nothing useful that the sidebar active state doesn't already provide.

### P2 — SectionCard is the only card primitive
`SectionCard` (border, padding, title, description, children) is used for:
- filter controls
- upload forms
- informational text lists
- data tables
- action panels
- settings forms
- nested lists inside other SectionCards (target-costs page)

There is no visual differentiation between "primary working area" and "secondary annotation". Everything looks equally important.

### P3 — Stat cards over-explain their own values
Every stat card has a `stat-copy` paragraph that describes what the stat means:
- "Последние загрузки" → "Свежие import runs и их текущий статус."
- "Alerts" → "Суммарное число сохранённых сигналов по текущей выборке import runs."
- "Глобальный target" → "Базовое значение для всех summary, если нет override."

An operator who logs in daily does not need these explanations. The number is the point. The description robs the stat card of visual punch and adds reading noise.

### P4 — Import run detail is an undifferentiated wall
The `/imports/[importRunId]` page has 15+ sections displayed one after another with equal visual weight:
metadata → parsing stats → operational diagnostics → historical base → route breakdown → skip reasons → analyzer summary → reason codes → comparison groups → analyzer results → alert events → telegram delivery → raw errors → normalized rows preview.

An operator opening an import run wants to know:
1. Did it succeed?
2. What alerts were generated?
3. Is delivery confirmed?

The full pipeline diagnostics should be collapsed / secondary.

### P5 — Navigation style wrong for the content
Sidebar nav items use the same card pattern as content cards (border, background, padding). They look like clickable blocks, not navigation. This makes the sidebar feel visually heavy and reduces the distinction between navigation space and content space.

### P6 — Navigation badges provide no operational value
"обзор", "план", "live", "база", "core", "url", "csv", "help", "admin" — these are taxonomy labels, not status signals. An internal operations tool sidebar should show counts that matter: pending imports, active campaigns, unresolved alerts.

### P7 — Language inconsistency
- Nav items: "Queue", "Active", "Scaling", "Creatives", "Approaches", "Landers" (English) alongside "Импорты", "Гид" (Russian)
- PageHeader eyebrow labels: "Creative details", "Imports", "Import Run" (English)
- Stat cards in Russian but with inline code backtick references: `` `TELEGRAM_ALERTS_ENABLED` ``
- Dashboard description: "без лишнего BI-шума" is fine; developer-facing "future recommendations" is not

### P8 — Admin pages contain operational data that belongs on the dashboard
The target costs admin page shows "Historical base" (cumulative spend, import coverage, top approaches by spend). This data belongs on the dashboard. The settings page should only contain settings.

### P9 — Creative detail shows all metadata fields regardless of population
The meta-list under "Медиа и внешнее хранение" lists all 10 URL fields even when most say "Не указан". A creative backed by Google Drive might have 3 filled fields and 7 empty ones — all rendered with equal visual weight.

### P10 — Gallery lacks operational controls
The creatives gallery has no filter bar, no search, no total count visible at top, no way to filter by approach. The mode switcher (Список/Галерея) is just two inline buttons — not a clear affordance.

## 3. Target UI Direction

**The product should move from "explanation-first" to "signal-first".**

Each page should answer a question the operator has when they arrive. The question, and the answer, should be visible immediately — within the first two scroll-free viewport lines of content.

**Proposed structural principles:**

1. **One title per page.** The topbar h1 = the page identity. The PageHeader inside the content frame goes away or merges with a lighter "page intro" strip. No eyebrow + h2 + description triple.

2. **Differentiate primary and secondary sections.** Primary working area = full-width, no border, flat. Secondary/explanatory = subdued panel (muted background, no border). Current SectionCard → use only for secondary content. Introduce a "working-area" wrapper without visual container chrome.

3. **Stat cards: number only, no paragraph.** The label + value is enough. Reserve `stat-copy` for cases where the value needs unit context (e.g., a date range). Delete self-explanatory descriptions.

4. **Filter/action bar as a first-class primitive.** Instead of wrapping filters in a SectionCard with a title ("Фильтры") and description, use a compact horizontal filter bar directly below the page header. This is what operational tools look like.

5. **Import run detail: progressive disclosure.** Show the critical summary (status, alert count, delivery status) in a compact header block. Everything below is detail that can be in collapsible sections or tabs.

6. **Sidebar: link style, not card style.** Remove the border/background from nav items. Use a simple hover highlight + active indicator. Reduce visual weight of the navigation frame.

7. **Gallery: filter bar at top.** The gallery page needs a compact filter row (approach select, status select) before the grouped content.

## 4. Page-by-Page Findings

### Shell (layout, sidebar, topbar)

**Topbar:**
- Renders the nav item title as an h1. Redundant with sidebar active state and page-level title.
- User chip is well-designed but takes a lot of horizontal space.
- Theme toggle + sign out are correct placement.
- **Fix:** Topbar should show breadcrumb (section / page) rather than re-stating the page title, or be removed in favor of in-page heading.

**Sidebar:**
- Good grouping: Операции / Библиотека / Система.
- Nav items as bordered cards = too heavy. Each nav item currently has `border`, `border-radius: 18px`, `background: var(--background-muted)`, and 11px/12px padding.
- The `pill` badge inside nav items adds no information.
- The "FB Ads Ops" brand lockup at top is clean and proportionate.
- **Fix:** Strip nav items to: left accent bar on active, simple hover highlight. Remove pill badges or replace with live counts.

### Dashboard (/)

**Good:**
- Date preset row (7d / 14d / 30d / Все время) is a clean filter pattern.
- Approaches-first layout before campaign table is the right hierarchy.
- Target cost status tones (good/neutral/warning/muted) are effective.

**Problems:**
- `SectionCard` wraps the date filter controls ("Срез"). The section card has a title "Срез" and a description. A date filter should not be a named section — it should be a filter bar.
- Summary grid has 7 stat cards, all with `stat-copy` paragraphs. "Подходы: 3 — Видны в historical overview" — the description adds zero value. "Кампании: 12 — Готовы для drill-down" — same.
- Approach cards: the "Открыть подходы" action button goes to `/approaches` (the full list), not to the specific approach. This is broken UX — a button inside an approach card should link to that approach's detail.
- Dashboard description says "без лишнего BI-шума" — but the 7-card stat grid and 3-column metric cards per approach is quite noisy.
- `SectionCard` for "Подходы" and "Кампании" is correct as a containment primitive, but the section headers ("Главный слой обзора: накопленный расход...") are too verbose.

### Imports (/imports)

**Good:**
- Upload form + pipeline info side by side is a reasonable layout.
- Import runs table is clean and informative.
- Status pills (Готово/Ошибка/pending) are correct.

**Problems:**
- "Коротко по потоку" section explains the storage path and API route. This is developer documentation, not operational UI. An operator does not need to see `uploadsRoot` path or know about `/api/imports/upload` on this screen. Move to `/guide` or remove.
- 4 stat cards all have `stat-copy` that re-describes the stat. "Последние загрузки: Свежие import runs и их текущий статус." — the label already says "Последние загрузки".
- The eyebrow "Imports" is English in a Russian UI.

### Import Run Detail (/imports/[id])

## 5. Recommended Shared UI System / Layout Language

Replace the current "blocks everywhere" model with a set of purposeful layout primitives:

### Primitive 1: Page frame
`content-frame` stays. But remove the `PageHeader` from inside it as a separate component. Instead, use:
- **Topbar:** section name (breadcrumb, not page title). Already exists but underused.
- **Page title + primary actions bar:** a single row at the top of content-frame: h1 (page-specific title) + action buttons right-aligned. This replaces the combination of topbar h1 + PageHeader eyebrow/title/description.

### Primitive 2: Summary strip (replaces stats-inline)
A compact horizontal strip of 3–5 key numbers. **No paragraph descriptions**. Label + number only. Rendered as a single background-muted row, not individual bordered cards.

```
[ Импортов: 12 ]  [ В работе: 1 ]  [ Ошибки: 0 ]  [ Alerts: 47 ]
```

### Primitive 3: Filter/action bar
A compact full-width row containing filter controls (selects, search input) left-aligned + primary/secondary actions right-aligned. **Not wrapped in a SectionCard with a title.** Used on: imports, creatives, gallery.

### Primitive 4: Working section
The primary content area. No visual container — just the content itself (table, form, grid). Has a `section-title` row (plain h3, no border) if labeling is needed.

### Primitive 5: Annotation panel
A subdued (muted background, subtle border) card for secondary info, descriptions, reference data. Used for: "what to do next", help notes, pipeline info. Replaces the "explanatory SectionCard" use case.

### Primitive 6: Settings section
For admin forms: a two-column layout with `h3` section heading + compact form below. No multiple levels of nested panels. Max one level deep.

### Primitive 7: Detail sidebar
On detail pages (creative, import run), a narrow right column (280px) showing timestamps, metadata, status facts. The left column holds the primary content (media preview, table, form). This pattern already exists via `hero-grid` but should be more systematic.

### Primitive 8: Collapsible diagnostic section
For import run details: a component that shows a heading + summary inline, expanding on click to reveal detail tables/lists. Key for reducing the wall-of-sections problem.
